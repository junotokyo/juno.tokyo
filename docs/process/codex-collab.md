# Codex CLI との協業ルール

このファイルは `CLAUDE.md` の「サブファイル参照ガイド」から trigger 時のみ Read されるサブファイル。
`codex` コマンドを実行する前に必ず本ファイルを Read してから進めること。

Claude が master、Codex CLI は道具。Codex の指摘は参考意見であり、**採否は Claude が決めてユーザーに
説明可能な形で示す**。

> このフレームワークは PopScan / Filmator で成熟させたものを juno.tokyo 向けに汎用化したもの。固有の
> プロファイル名・テストコマンドは `PROJECT.md` の値に従う。

## フェーズ分担マップ（0〜10）

プロダクト開発を 0〜10 で捉えたときの Claude / Codex 分担。数字は目安。

| 区間 | 工程 | 主導 | 補助・検証 | 参照 |
|------|------|------|-----------|------|
| 0〜3 | 新機能・機能改善のプランニング | **Claude** | — | （通常フロー） |
| 3〜5 | プランのブラッシュアップ | Codex レビュー | **Claude が責任判断** | セクション A / A-2 |
| 5〜8 | 実装 | **Claude 実装** | Codex 実装後レビュー | セクション B |
| 8〜10 | 不具合・改善要望の調査・改善案 | **Codex 主導調査** | **Claude がレビュー・責任判断** | セクション F |

- AI サイドの責任者は常に **Claude**（採否を決めユーザーに説明可能にする）。最終成果物の責任者は **ユーザー**。
- 8〜10 を Codex 主導にする狙い: 実装した Claude は実装時の mental model に引っ張られ、不具合・改善時にその枠から
  抜けにくい（anchoring bias）。実装文脈ゼロの Codex に独立 framing を取らせ、Claude は客観材料を得た上で責任判断する。

## 可視性ルール（必須）

ユーザーが Codex への委譲を目視できるよう、`codex` コマンドを実行する**直前**に 1 文の宣言を出すこと。

- 形式: 「〜のため Codex を呼びます（`codex <subcommand> --profile <name>` 等）」
- 応答が返ったら、採用 / 却下 / 保留の分類結果を 1〜2 行でユーザーに要約報告する

## 節約方針

Codex（Business）は上限が先に来やすい一方、Claude（Max）は余裕あり。**必要最小限**に呼ぶ：

- **回数を絞る**：プランレビューはユーザー承認前に 1 回、実装後レビューはコミット前に 1 回が原則（セクション A/B）。
  難航時の原因究明（E-1/F-1）も**既定 1 回**。同一案件で何度も投げない。
- **範囲を絞る**：巨大 diff を丸投げせず、論点を明確にした stdin プロンプト（Q1〜N 形式・「簡潔に・長文不要」を明記）で要点だけ問う。
  変更が小さい/自明な採否反映は再レビューしない（採否を 1 行明記してスキップ）。
- **渡す量を絞る**：プランレビューはプラン全文ではなく、**レビュー観点（テスト設計・本番影響・セキュリティ等）で実際に問う部分だけ**を
  抜粋して渡す（背景・既決事項・無関係セクションは省く）。
- **Codex が読み回る範囲を必要十分に絞る**：実装後レビューで変更が広域（目安: **5 ファイル超 or 300 行超**）なら
  `codex review` 全体ではなく、レビュー観点上重要な**ファイルに `codex exec` で絞る**。プロンプトでも「対象は〇〇に限定・
  周辺探索は不要」と読む範囲を明示する。
- handoff・typo・1 ファイル数行・既存パターン踏襲はレビュー対象外（セクション A の「スキップ可」を積極活用）。

### Claude 側 effort を上げて Codex 往復を減らす

Codex 消費の主因は「指摘→修正→再レビュー」の往復。**Claude Code の工数（effort）を「高(high)」→「xhigh」に上げ、
プラン・実装の初回完成度を高めることで Codex のレビュー往復回数を減らす**戦略を取る（「Claude のトークンで Codex の
トークンを買う」交換）。

- 段階は **xhigh から**。max への直行は非推奨。
- 効くフェーズは **ExitPlanMode 前のプラン精度** と **セクション B 前の実装初回品質**。

## 共通: サブコマンド別の必須オプション

`codex exec` を呼ぶときの必須オプション：

- `--profile juno-tokyo-review`（read-only）または `--profile juno-tokyo-edit`（workspace-write）
  - プロファイルが未設定なら `PROJECT.md`「Claude 委任 Codex CLI プロファイル」のセットアップに従う
- `--cd "$PWD"`（worktree 事故防止のため必須）

`codex review` は `--profile` / `--cd` フラグを**受け付けない**（top-level オプションは `-c/--config`,
`--uncommitted`, `--base`, `--commit`, `--enable`, `--disable`, `--title` のみ）。代わりに：

- カレントディレクトリを `--cd` 代わりに使うので、**呼び出し前に `cd "$PWD"` 相当の状態を確認**する（git worktree のルート）
- sandbox / approval は review コマンドが内部で read-only モードに固定するため指定不要
- profile 相当の設定上書きが必要なら `-c <key>=<value>` で個別指定する

長文プロンプトは引数渡しで文字化け hang する事象を確認済。**stdin 経由（`codex exec -`）で渡す**こと。

## A. プランレビュー（非自明なプランのみ）

### 正準フロー（順序厳守）

```
Claude プラン作成
  → Codex プランレビュー（非自明）→ 採否分類してプランへ反映
  → ExitPlanMode：Codex 反映済みプラン（採否サマリ込み）をユーザーに提示
  → ユーザー承認 / 方針指示
      ├─ 無修正承認 → そのまま実装へ
      └─ 方針変更あり → 変更後プランを 2 回目 Codex レビュー（A-2）→ 実装へ
  → 実装
  → Codex 実装後レビュー（コミット前・セクション B）→ 採否分類 → コミット → main FF
```

**Codex プランレビューはユーザー承認の「前」に必ず実施する。** ExitPlanMode で提示するプラン本体は
**Codex 採否分類（採用 / 却下 / 保留）を可視で含む**こと。ユーザーは pre-Codex プランではなく
**post-Codex プラン**を承認する。

ExitPlanMode を呼ぶ**直前**に、以下のいずれかを満たすプランは Codex レビューを通す：

- 複数ファイルにまたがる変更
- `api/`（Vercel Functions）・`middleware.js`・`vercel.json` に手を入れる
- 公開 URL・admin 認可・KV キー設計など本番影響の大きい領域
- 自動テストでカバーしづらく、検証手段が不確かな実装
- 新規 abstraction の導入・既存アーキテクチャの変更

該当しない場合（typo・1 ファイル数行・既存パターン踏襲等）はスキップ可。**判断理由を 1 行で明示する**。

```bash
PLAN=/Users/jokamoto/.claude/plans/<plan-file>.md
{ cat "$PLAN"
  printf '\n\n---\n\n'
  cat <<'EOF'
上記は Claude が作成した実装プランです。Critical にレビューしてください。
特に注目してほしい観点:
- テスト設計の不足（新ロジック対応テスト、エッジケース漏れ、モックで隠れる実挙動の明示）
- 本番影響の見落とし（push＝Vercel auto deploy、admin Basic-auth の解除、middleware の副作用、KV 上書き）
- セキュリティ（admin トークン露出、Basic-auth bypass、X-Robots-Tag の崩れ）
- 抜け・矛盾・前提の誤り

Q1〜N の形で簡潔に。長文の解説は不要。
EOF
} | codex exec --profile juno-tokyo-review --cd "$PWD" -
```

> 注意: `cat plan.md | codex exec - <<'EOF' ... EOF` のように書くと heredoc が stdin を上書きし、プラン本文が
> Codex に渡らない。**プラン本文＋指示を 1 つの stdin ストリームに結合**する上記パターンを使うこと。

レビュー結果は Claude が **採用 / 却下 / 保留** を明示してプランへ反映してから ExitPlanMode を呼ぶ。Codex の指摘を盲従しない。

## A-2. 承認後の方針変更時の再レビュー（条件付き）

ExitPlanMode で提示した（Codex 反映済み）プランに対し、ユーザーが承認時に **実装方針を実質的に変更する指示**を
出した場合のみ、実装着手前に再度プランレビューを通す：

- **トリガー**: 承認コメントがアプローチ・設計・スコープを実質的に変える（例:「別アプローチで」「この機能は外して」
  「順序を変えて」）。軽微な文言修正・スコープ確認の質問・「OK そのまま」は**対象外**。
- **アクション**: 変更後プランを再度セクション A の `codex exec --profile juno-tokyo-review --cd "$PWD" -` に通し、採否分類してから実装着手。
- **無修正承認時**: 同一プランの再レビューは**省略**（「ユーザー無修正承認のため A-2 スキップ」と 1 行明示）。

## B. 実装後レビュー（全実装で実施、コミット前）

実装完了後・コミット前に必ず以下を実行：

```bash
codex review --uncommitted   # 未コミット変更（staged + unstaged + untracked）をレビュー
# あるいは:
codex review --base main     # main からの差分を ALL コミット込みでレビュー（排他: --uncommitted と併用不可）
```

巨大 diff だと指摘が浅くなる傾向があるため、必要に応じてファイル単位で `codex exec` に切り替えてよい。

Claude は出力を読んで、各指摘を **採用 / 却下 / 保留** に分類してユーザーに報告：

- **採用**: 反映してから commit
- **却下**: 理由を 1 行で（誤解・既知の制約・代替案あり 等）
- **保留**: 別 issue / 次セッションへの申し送りに回す
- **Codex の指摘が薄い/見落としあり**: Claude 側でテスト追加・追加修正

テストの実行は **Claude 側で行う**（`PROJECT.md` のテストコマンド参照。Codex CLI に本番／Preview 検証能力はない）。

## C. コード修正の代行（限定的、ユーザー承認必須）

以下のいずれかに該当する変更で、Codex に書き換えさせると効率が良いと Claude が判断したとき、**ユーザーに
具体的な依頼内容を提示して承認を得た上で**実行する：

- 機械的な置換・rename・大量のフォーマット修正
- Codex のレビュー指摘を Codex 自身に反映させた方が早いケース
- Claude が混雑している領域と独立した、明確に切り出せるサブタスク

呼び出し前に `pwd` と `git worktree list` の出力をユーザーに見せて承認を取る（worktree 外への誤編集防止）：

```bash
codex exec --profile juno-tokyo-edit --cd "$PWD" - <<'EOF'
<具体的・限定的な変更指示。スコープ外を触らないこと、commit/push しないことを明記>
EOF
```

実行後 Claude が `git diff` でレビューし、必要なら手動修正・テスト実行・コミットを行う。Codex に commit・push はさせない。

## D. 関係性の明示

- Claude が master、Codex は道具
- 書き込み権限付与（`--profile juno-tokyo-edit` / `--sandbox workspace-write`）はユーザー承認の上のみ
- Codex の判断に責任を委ねない（採否は Claude が決め、ユーザーに説明可能な状態にする）

## E. 独立調査としての活用と見解相違の扱い（難航時 必須）

> Codex を「自分の解法前提の査読者」にしか使わず迷走した失敗の再発防止。
> 不具合・改善要望の**入口**では本セクションを「難航時」に限らず**既定発動**する（セクション F が E を参照する機構）。

Codex は「Claude の枠内プランの査読者」に閉じない。**種類の違う AI** として独立した framing を引き出し、
より俯瞰した客観視点で正しい方向を選ぶために使う。

### E-1. 難航/非自明案件は「原因究明（解を渡さない）」を先に1回挟む

- 非自明バグ・回帰・行き詰まり案件では、**自分の解法/スコープを Codex に渡す前に**、まず「根本原因と正しい
  スコープは何か」を**解を提示せず**独立に問う（`codex exec --profile juno-tokyo-review --cd "$PWD" -`）。
  自分の解を先に見せると Codex も枠内批評になり独立性が失われる。
- その独立見解を得てから解法プラン化し、改めてセクション A のプランレビューへ。

### E-2. 見解相違は「採用/却下の項目処理」で終わらせない

- Codex の framing が自分と実質的に異なる、または Codex が前提への疑義・警告を出した場合、それを単なる採否項目に
  せず**相違自体を論点化**する。
- Codex の警告を「採用したが据置」で流さない。**前提を疑う信号**として扱い、必要なら E-1 の独立原因究明へ戻る。
- Claude と Codex で方向が割れたまま収束しないときは、両者の見解差をユーザーに俯瞰提示して判断を仰ぐ（黙って自分案を通さない）。

### E-3. ループ時は「前提反証」を明示依頼

- 同一サブシステムで自分の修正/ユーザー差し戻しが **2 回** 起きたら、次の patch に入る前に Codex へ「前提を疑え・
  別アプローチ・より単純な不変条件で代替できないか」を**明示的に**依頼する（前提追認でなく前提反証を求める）。
- **Claude 前段（前倒し）はセクション G のループブレーク**＝失敗 2 回で先に cold-start サブエージェント（解を渡さない）に
  かけ、収束しなければ Codex へ。3 回目を anchoring のまま当てない。

### E-4. 対症療法を Codex に追認させない

- Codex が最小 patch を勧めても、それが原因を覆い隠す対症療法なら、「症状除去」でなく「原因除去（構造的整合）」の
  観点で Codex に再評価させる。

## F. 不具合・改善要望は Codex 主導調査が既定（8〜10 区間）

> セクション E の運用を、難航時に限らず**不具合・改善要望の入口で既定発動**する形に格上げしたもの。
> `CLAUDE.md` Trigger 4 がこのセクションへの入口。

実装者(Claude)は anchoring bias を持つ。バグ時点で参照する mental model は、まさにその不具合を予測できなかった
枠組みである。同じ枠で調査を続けると抜けられない。よって不具合・改善要望は **Claude が自分で原因調査・改善案検討に
直行せず、まず Codex に独立調査させる**を既定とする。

### F-1. 不具合（バグ・回帰）

- Codex は cold-start で良い（実装文脈ゼロの fresh eyes が肝）。
- **解・スコープを渡さず**「根本原因と正しいスコープは何か」を独立に問う。E-1 と同じ呼び出しを、**難航時に
  限らず既定で 1 回**実施する。
- Claude は Codex の調査結果をレビューし、採用 / 却下 / 保留を**責任判断**してプラン化 → 以降セクション A の通常フローへ。

### F-2. 改善要望

- 改善要望はプロダクト意図・ロードマップ知識を要するため、Codex の白紙投入は方向と衝突しやすい。
- **Claude が先に枠付け**する: 意図 / 制約 / 非ゴール / UX 目標を明文化 → その枠を渡して
  **Codex が選択肢を独立調査**（複数案・トレードオフ）→ Claude がレビュー・責任判断。

### F-3. Claude のレビュー責任（2 つの失敗モードを防ぐ）

- 失敗モード: (a) 鵜呑み、(b) 自分が書いたコードへの anchoring で正しい Codex 診断を却下。**(b) はこの分担が潰したい現象の再来**。
- **Codex の診断に Claude が反対したら、それ自体を「前提を疑うトリガー」にする**（Claude 優位のタイブレークにしない）。

### F-4. 収束しないときのユーザー提示フォーマット

Claude と Codex で方向が割れて収束しない場合、深い技術論点をそのまま投げず、以下の判断軸で**比較表**にしてユーザーに上げる:

| 観点 | 案 A | 案 B |
|------|------|------|
| 技術的難易度・工数 | | |
| メンテナンス性 | | |
| UX 面の比較 | | |
| その他トレードオフ | | |

- **前提: UX 最優先**。プロダクトはユーザーのためにある。良い UX を満たすことを前提に、開発難易度・メンテナンス性を比較する。
- 技術的な深掘りは要点を平易に 1〜2 行添える程度に留め、判断軸の比較を主とする。

### F-5. 所有権

Codex 主導フェーズでも commit / main FF（＋push）/ Handoff 更新は **Claude**（PROJECT.md 既定）。
Codex に commit・push はさせない。

## G. Claude セルフレビュー（サブエージェント）— Codex レビューの前段

Codex は "種類の違う AI" で多様性が高く指摘の質が高い（実証）。Claude サブエージェント（`Agent` ツール）は
**同じ Claude＝盲点が相関する**が、**別コンテキスト＝自分の固着した枠を共有しない**ため客観性が出る。

- **位置づけ / 順序**: Claude サブエージェント＝**安価・高速な一次防衛＋anchoring 回避**。これで完成度を上げ、
  **その上で** Codex（A/B）へ＝Codex は**多様性バックストップ**（深い/重大/繰り返し失敗）。Codex のトークンを温存する。
- **使い分け**:
  - 通常実装の完成度上げ・自明寄りの確認 → **Claude サブエージェント**で自己レビュー（Codex を投げる前に）。
  - 本番影響の大きい領域（admin 認可・KV キー設計・middleware・vercel.json）・セキュリティ・**繰り返し失敗**
    → 多様性が要るので **Codex**（サブエージェントだけで済ませない）。
- **やり方（Codex F-1 と同型＝自分の解を渡さない）**:
  - **G-1 cold-start 原因調査**: 症状＋該当ファイルのみ渡し、自分の仮説は伏せて「根本原因と直し方」を独立に問う。
  - **G-2 敵対的レビュー**: 「この実装が**仕様に反する入力を 1 つ見つけろ**」と反証を依頼し、**全入力ケースを列挙してトレース**させる。
  - **G-3 完成度クリティック**: 「抜け（未トレースのケース・未検証の主張・テスト漏れ）は何か」。
- 🔴 **ループブレーク（E-3 の Claude 前段・前倒し）**: 同一箇所で**失敗 2 回**で即パッチを止め、まず **G-1 cold-start
  サブエージェント**（解を渡さない）→ それでも収束しなければ Codex。実装者 anchoring のまま 3 回目を当てない。
- 🔴 **完了宣言の前に「全入力ケースをトレース」**: HTTP メソッド分岐・認可分岐・KV キー分岐・タイムゾーン境界等は、
  報告されたケースだけでなく**全入力ケースを列挙しコードを 1 ステップずつ追って仕様と照合**する。
- 🔴 **対症療法・band-aid 禁止（E-4 再徹底）**: 「修正が要件を削る/変える」ならそれは修正ではない。症状を消すために
  仕様を変えたら必ず止めてユーザーに報告する。
