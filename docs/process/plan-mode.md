# プランモード — ExitPlanMode 前の自己レビュー

このファイルは `CLAUDE.md` の「サブファイル参照ガイド」から trigger 時のみ Read されるサブファイル。
プランモードに入ったら Phase 1 開始時に必ず本ファイルを Read してから計画を組み立てること。

ExitPlanMode を呼ぶ前に、以下 6 項目を **必ず** 確認すること。確認したことをテキストで明示する。

## ① セキュリティ
- 内部ツール・スクリプト・設定ファイルが意図せず公開される経路がないか
- 認証情報・トークン・秘密情報が露出しないか（`KV_REST_API_TOKEN` / `ADMIN_BASIC_PASS` 等）
- `middleware.js` の Basic-auth ゲートを意図せず外していないか（`/popscan/admin/*` と admin API）
- `X-Robots-Tag: noindex, nofollow` を admin に保てているか

## ② コスト・副作用（本番影響）
- 🔴 **`git push origin main` は Vercel が即 Production にデプロイ**。確信のない変更を push しない
- 意図しない API 呼び出し・KV 書き込み・課金が発生しないか
- 「commit/push → 自動デプロイ」のような連鎖を見落としていないか
- Preview デプロイで確認できる変更は、まず Preview を踏んでから main へ
- DNS / Mail レコードへの影響がないか（基本ノータッチだが念のため）

## ③ 事実の正確性
- 「〜は起きない」「〜は安全」等の断言は、実際に確認したことか
- 思い込み・推測を断定的に述べていないか（不確かなら「確認していないが〜と思われる」と明示、または確認してから述べる）
- DNS / Mail 等は `dig` で実測したかどうかを明示

## ④ 影響範囲
- 計画が間違っていた場合の最悪ケースは何か
- 取り返しのつかない操作が含まれていないか（KV の特定キー削除・admin パスワード再設定・DNS 変更等）
- 公開 URL（特に `/popscan/`）の構造を破壊していないか

## ⑤ テスト設計
プランの「検証方法」セクションに以下を必ず明記すること：
- **自動テストでカバーするケース**（具体的なテスト関数名・検証内容）
- **自動テストできない範囲**（理由も含めて記載）とその手動確認方法（Preview デプロイで何を踏むか・どのページを開くか）
- 「既存テストが通ること」だけを検証方法として記載してはならない。新実装に対応する新テストを計画する
- 🔴 **全入力ケースのトレース**: 対話/UI/状態ロジックは、**全入力ケースを列挙**し（例: 未認証/認証済み、空 KV/既存値、
  タイムゾーン境界、HTTP メソッド、エラーパス）、各ケースをコードに沿って 1 ステップずつ追って仕様と照合する計画にする。
  報告/想定ケースだけ確認して「OK」としない。

**モック/外部依存を使う場合の追加チェック**：
外部 API・KV・middleware をモックする場合、以下を明記すること：
- 「このモックテストが証明すること」（ロジックの何を検証するか）
- 「このモックテストが証明しないこと」（実 API / 実 KV / 本番 middleware が代替している部分）
- 実挙動の手動確認手順（具体的なステップ・Preview デプロイ URL の踏み方）

「テスト通過＝動作確認完了」とユーザーに報告してはならない。

## ⑥ Codex レビュー判定
- 本プランは非自明トリガー（`docs/process/codex-collab.md` セクション A）に該当するか？
- 該当するなら Codex review を**ユーザー承認の前に**実施したか、各指摘を採用 / 却下 / 保留のどれに分類したかを明記してから ExitPlanMode を呼ぶ
- **ExitPlanMode で提示するプラン本体に「Codex レビュー反映」節（Codex 指摘と採用 / 却下 / 保留分類）を必ず含める**。ユーザーは pre-Codex ではなく post-Codex プランを承認する
- 該当しないと判断した場合、その理由を 1 行で明示
- ユーザーが承認時に**実装方針を実質的に変更する指示**を出した場合は、実装着手前に変更後プランを再度 Codex レビュー（`codex-collab.md` A-2）。無修正承認なら省略（1 行明示）
